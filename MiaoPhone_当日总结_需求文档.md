# Memory Lab · 当日总结功能需求

> 在 Memory Lab 手动测试 tab 中新增「当日总结」模式，用于测试和迭代当日记忆总结的提示词效果。
> 本功能是测试阶段，暂不注入聊天上下文，纯手动触发 + 本地存储。

---

## 一、手动测试 tab 改造

### 1.1 模式切换

在手动测试 tab 顶部新增 segment control（两个切换按钮）：

- **叙事元提取**（默认，即现有功能，保持不变）
- **当日总结**（新增）

切换时，下方的「导入对话」区和「结果」区跟随变化，两个模式完全独立互不干扰。

### 1.2 当日总结模式 - 导入对话区

切换到「当日总结」模式后，导入区变为：

- **日期选择器**：选择一个日期
- **拉取按钮**：点击后从当前助手的微信聊天记录中，拉取该日期当天的所有消息
  - 按消息的时间戳筛选（当天 00:00:00 ~ 23:59:59）
  - 拉取后在预览区显示消息条数和时间范围
  - 如果当天无消息，提示「该日期无聊天记录」
- **预览区**：显示拉取到的对话预览（复用现有的对话预览组件样式）

### 1.3 当日总结模式 - 生成区

- **按钮**：「生成当日总结」（替代原来的「提取叙事元」按钮）
- 点击后调用总结模型，将拉取到的当日消息 + 当日总结提示词发送给 API
- 等待返回后，解析 JSON 结果并渲染

### 1.4 结果渲染

返回的 JSON 结构为：

```json
{
  "date": "YYYY-MM-DD",
  "daily_memory": "第一人称的当日记忆文本...",
  "observations": [
    { "content": "观察条目1..." },
    { "content": "观察条目2..." }
  ]
}
```

渲染为卡片形式，分两块展示：

**daily_memory 区域：**
- 标题「当日记忆」
- 正文渲染 daily_memory 字段的文本内容

**observations 区域：**
- 标题「观察区」
- 每条 observation 渲染为独立的小卡片或列表项

卡片底部显示：
- 生成时间戳
- 使用的模型名称
- 「复制 JSON」按钮（复制完整的原始 JSON）

### 1.5 历史结果对比

**核心需求：提示词迭代过程中需要对同一天的数据反复测试、对比结果。**

- 每次生成的结果不覆盖之前的，而是追加到结果列表中
- 结果按生成时间倒序排列（最新的在上面）
- 每条结果卡片显示生成时间戳，方便区分不同轮次
- 提供「清空结果」按钮，一键清空当前所有测试结果

测试结果只在当前会话中保留（存内存即可），退出 Memory Lab 页面后丢弃。这些是临时测试数据，不需要持久化。

## 二、提示词配置

### 2.1 配置位置

在配置 tab 中，现有「提取提示词」区域的**下方**，新增一个区域：

- 标题：「当日总结提示词」
- 一个多行文本编辑框，用于编辑当日总结的提示词
- 保存到 localStorage（与叙事元提示词同一个配置 key `miaomiao_mlab_config` 中，新增字段 `dailySummaryPrompt`）

### 2.2 编辑框行为

与叙事元提取提示词一致：
- 编辑框初始为**空**，由用户自行填入提示词
- 未填写时点击「生成当日总结」应提示「请先在配置中填写当日总结提示词」
- 用户填写后自动保存到 localStorage

### 2.3 提示词中的变量

提示词中 `{date}` 在生成时自动替换为用户选择的日期（格式 `YYYY/M/D`，与现有长期记忆保持一致）。

## 三、总结模型调用

### 3.1 模型选择

复用配置 tab 中已有的「总结模型」选择器（即叙事元提取用的同一个模型选择器），当日总结和叙事元提取共用同一个模型配置。

### 3.2 请求构建

```javascript
messages: [
  {
    role: "user",
    content: dailySummaryPrompt（替换{date}后） + "\n\n---\n\n以下是聊天记录：\n\n" + 格式化后的当日消息文本
  }
]
```

消息文本格式化方式：与叙事元提取一致，用 `appData.settings.userName` 和当前助手名标记发言者。

### 3.3 Temperature

复用配置 tab 中已有的 Temperature 滑块。

### 3.4 JSON 解析

与叙事元提取一致的 JSON 解析逻辑：
- 尝试直接 `JSON.parse`
- 失败则尝试提取 ```json``` 代码块后再解析
- 解析失败则显示错误信息和原始返回文本

## 四、当日总结存储（正式存储，非测试结果）

### 4.1 说明

手动测试区的对比结果只存内存（临时），但当用户确认某条总结质量OK后，需要有一个「保存」按钮将其存为正式的当日总结。

### 4.2 存储位置

IndexedDB，存储在现有的数据结构中。

建议在 `appData` 中新增字段 `dailySummaries`：

```javascript
appData.dailySummaries = {
  "2026-03-10": {
    date: "2026-03-10",
    daily_memory: "...",
    observations: [{ content: "..." }],
    model: "claude-opus-...",
    created_at: "2026-03-11T02:30:00Z",
    updated_at: "2026-03-11T02:30:00Z"
  },
  // ...
}
```

### 4.3 保存操作

每条测试结果卡片上增加一个「保存为正式总结」按钮：
- 点击后将该条结果存入 `appData.dailySummaries[date]`
- 如果该日期已有正式总结，提示「该日期已有总结，是否覆盖？」
- 保存成功后 toast 提示

### 4.4 云端同步

正式总结需要同步到 Supabase，遵循现有的脏标记机制：
- Supabase key：`daily_summaries`（整个对象作为一个 key 存储）
- 保存时标记 `_cloudSyncDirty`

## 五、UI 细节

### 5.1 样式

新增部分的 UI 风格与 Memory Lab 现有风格保持一致（紫色渐变主题、圆角卡片、白色背景区块）。

### 5.2 segment control 样式

参考 iOS 风格的 segmented control：
- 两个选项并排
- 选中项有紫色背景 + 白色文字
- 未选中项为透明背景 + 紫色文字
- 放在手动测试 tab 的最顶部，导入区域之上

### 5.3 日期选择器

使用原生 `<input type="date">`，样式与 Memory Lab 整体风格适配。默认值为昨天的日期。

### 5.4 结果卡片

- daily_memory 和 observations 用不同的视觉区分（如 observations 区域用更浅的背景色或虚线边框）
- 每条 observation 前可以加一个小圆点作为列表标记
- 「复制 JSON」「保存为正式总结」按钮放在卡片底部，小号样式，不要太抢眼

---

## 六、不做的事情（明确排除）

- ❌ 暂不做自动触发（每天自动跑总结）
- ❌ 暂不注入到微信聊天上下文
- ❌ 暂不做批量日期范围总结
- ❌ 暂不做 observations 的分类标签
- ❌ 测试结果不持久化到 IndexedDB（只有「保存为正式总结」才写入存储）
