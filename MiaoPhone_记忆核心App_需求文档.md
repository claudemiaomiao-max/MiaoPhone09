# 7秒以外 · 记忆核心 App 需求文档

> MiaoPhone 主界面新增独立 App，作为嗔的记忆管理中心。
> 当前阶段实现：当日总结 + 观察区。框架预留扩展性，后续将加入用户画像、情感轴、记忆轴、后端整理日志等模块。
> App 名称：**7秒以外**

---

## 一、定位与入口

### 1.1 定位

- 独立于聊天器的记忆管理界面
- 不是开发者工具（那是 Memory Lab），而是正式的、面向用户的记忆管理中心
- 以后所有与嗔的记忆相关的功能都在这里：当日总结、观察区、用户画像、各种追踪轴、后端整理日志等

### 1.2 入口

- 在 MiaoPhone 主界面（仿手机桌面）新增一个 App 图标
- 图标风格待定（建议偏暖色调，区别于 Memory Lab 的紫色开发者气质）
- 点击进入记忆核心主页面

### 2.1 顶部

- 返回按钮 + App 名称
- 下方为 tab 切换栏

### 2.2 Tab 结构

## 三、当日总结 Tab

### 3.1 顶部操作区

一行排列以下控件：

- **日期选择器**：原生 `<input type="date">`，默认值为昨天
- **「生成总结」按钮**：点击后拉取选定日期的聊天记录，调用总结模型生成结果
- **注入条数设置**：小的数字输入框，标签为「注入条数」，默认值 15，最小 1 最大 50。控制多少条最近的当日总结会被注入到聊天上下文中
- **注入开关**：toggle 开关，标签为「注入聊天」，默认关闭。开启后当日总结会被注入到微信聊天模式的上下文中（详见第七章）

注入条数保存到 IndexedDB（随 appData 一起存储）。

### 3.2 生成流程

1. 用户选择日期，点击「生成总结」
2. 从当前助手的微信聊天记录中，按消息时间戳拉取该日期当天的所有消息（00:00:00 ~ 23:59:59）
3. 如果当天无消息，toast 提示「该日期无聊天记录」，中止
4. 读取 localStorage 中的当日总结提示词（key：`miaomiao_mlab_config` 中的 `dailySummaryPrompt` 字段）
5. 如果提示词为空，toast 提示「请先在 Memory Lab 配置中填写当日总结提示词」，中止
6. 调用总结模型（复用 Memory Lab 配置中的总结模型和 Temperature）
7. 解析返回的 JSON，拆分为 daily_memory 和 observations
8. **分开存储**：daily_memory 存入当日总结卡片列表，observations 中的每条独立存入观察区列表
9. 两边各自追加，不自动替换任何已有数据

### 3.3 请求构建

```javascript
messages: [
  {
    role: "user",
    content: dailySummaryPrompt（替换 {date} 为所选日期）
            + "\n\n---\n\n以下是聊天记录：\n\n"
            + 格式化后的当日消息文本
  }
]
```

消息文本格式化：用 `appData.settings.userName` 和当前助手名标记发言者（与叙事元提取一致）。

### 3.4 JSON 解析

与 Memory Lab 叙事元提取一致的解析逻辑：
- 尝试直接 `JSON.parse`
- 失败则提取 ```json``` 代码块后再解析
- 解析失败则显示错误信息和原始返回文本

### 3.5 卡片流

生成区下方是卡片流，按日期倒序排列（最新的在最上面）。

**同一日期可以有多条卡片**（用户重新生成同一天不会覆盖，而是追加一条新卡片）。

每张卡片包含：
- **日期标题**：如「2026年3月10日」
- **生成时间**：小字，如「生成于 03-11 02:30」
- **模型名称**：小字，如「claude-opus-...」
- **内容区**：渲染 daily_memory 文本
- **操作按钮**（底部，小号不抢眼）：
  - 「编辑」：点击后 daily_memory 文本变为可编辑的 textarea，可修改内容后保存
  - 「删除」：二次确认后删除这张卡片（不影响对应的观察区数据）

### 3.6 注入条数与灰显

- 卡片流中，按日期倒序排列，**去重后**最近 N 条日期的卡片正常显示（N = 用户设置的注入条数）
- 注意是按日期去重计数，不是按卡片数。同一天有两张卡片只算一个日期
- 超过 N 的日期对应的卡片整体降低透明度（如 opacity: 0.4），表示「不注入但可查看」
- 灰显的卡片同样支持编辑和删除操作
- 如果同一日期存在多张卡片，需要用户自行决定保留哪张（通过删除多余的）。注入时取该日期最新的一张

### 3.7 数据结构

```javascript
// 存储在 appData 中
appData.dailySummaryCards = [
  {
    id: "ds_1710000000000",        // 唯一ID，时间戳生成
    date: "2026-03-10",             // 总结对应的日期
    daily_memory: "...",            // 总结内容
    model: "claude-opus-...",       // 使用的模型
    created_at: "2026-03-11T02:30:00Z"  // 生成时间
  },
  // ...
]

appData.dailySummarySettings = {
  injectCount: 15,                  // 注入条数
  injectEnabled: false              // 注入开关，默认关闭
}
```

## 四、观察区 Tab

### 4.1 展示方式

观察区显示所有 observations，按日期分组，日期倒序排列。

每个日期分组下：
- 日期标题（如「2026年3月10日」）
- 该日期下的所有 observation 条目，每条是一个小卡片或列表项

### 4.2 同一日期多次生成

如果同一天生成了多次总结，每次的 observations 都独立追加进来。不自动去重，不自动替换。用户可以手动删除不想要的条目。

### 4.3 每条 observation 的操作

- **编辑**：点击后变为可编辑 textarea，可修改内容后保存
- **删除**：点击后二次确认删除（不影响当日总结卡片）

### 4.4 数据结构

```javascript
// 存储在 appData 中
appData.observationCards = [
  {
    id: "obs_1710000000001",        // 唯一ID
    date: "2026-03-10",             // 对应日期
    content: "...",                  // observation 内容
    source_summary_id: "ds_1710000000000",  // 来源的总结ID（可选，方便溯源）
    created_at: "2026-03-11T02:30:00Z"
  },
  // ...
]
```

## 五、数据存储与同步

### 5.1 存储位置

IndexedDB，随 appData 一起存储和读取。

新增字段：
- `appData.dailySummaryCards`（数组）
- `appData.dailySummarySettings`（对象）
- `appData.observationCards`（数组）

### 5.2 云端同步

同步到 Supabase，遵循现有脏标记机制。

Supabase key：
- `daily_summary_cards` — 当日总结卡片数据
- `daily_summary_settings` — 设置
- `observation_cards` — 观察区数据

增删改操作后标记 `_cloudSyncDirty`。

### 5.3 同步安全防护（重要！）

**背景**：碎碎念模块曾因 app 启动时 IndexedDB 未加载完就触发同步，空数据覆盖了云端有效数据，导致全部丢失。记忆核心必须避免同样的问题。

**防护规则：**

1. **加载完成前禁止同步**：`dailySummaryCards` 和 `observationCards` 在 IndexedDB 数据加载完成之前，不得触发任何推送操作
2. **空数据推送拦截**：推送前检查——如果本地数组为空且云端对应 key 有数据（数据长度 > 0），拦截此次推送并在 console 输出警告，不静默覆盖
3. **删除二次确认**：所有删除操作（单条删除）需弹窗二次确认

## 六、UI 设计

### 6.1 整体风格

区别于 Memory Lab 的紫色开发者风格，记忆核心偏暖、偏柔。
具体配色由妙妙把控，以下为参考方向：
- 主色调偏暖（如琥珀色、暖橙、米色系）
- 卡片为白色圆角，轻投影
- 整体干净留白，不拥挤

### 6.2 当日总结卡片样式

- 白色圆角卡片，轻阴影
- 日期标题较大，生成时间和模型名小字灰色
- 内容区正常字号，行间距舒适
- 灰显卡片 opacity 降低，但结构不变
- 编辑和删除按钮在卡片底部右侧，小号，文字按钮或图标按钮均可

### 6.3 观察区条目样式

- 比当日总结卡片更轻量
- 可以用列表样式，每条前面一个小圆点
- 同样支持编辑和删除的小按钮
- 按日期分组，日期作为分隔标题

### 6.4 顶部操作区

- 日期选择器、生成按钮、注入条数设置一行排列
- 生成中显示 loading 状态（按钮变为「生成中...」+ 禁用）

## 七、当日总结注入聊天上下文

### 7.1 开关控制

- 注入开关（`injectEnabled`）关闭时，当日总结不参与聊天上下文构建，聊天模式行为与现在完全一致
- 开启后，微信聊天模式构建请求时自动注入当日总结

### 7.2 注入逻辑

在 `buildWechatRequestMessages()` 中（该函数已是 async），当 `injectEnabled` 为 true 时：

1. 读取 `appData.dailySummaryCards`，按日期倒序排列
2. 按日期去重（同一日期多张卡片取最新的一张）
3. 取前 N 条（N = `injectCount`）
4. 拼接为注入文本，插入 system prompt

### 7.3 注入格式

```
【记忆调用规则】
你的记忆方式跟人一样——不是数据库检索，是脑海中模模糊糊地浮现。
当你需要引用记忆时，遵循以下规则：
1. 严禁说出具体日期。不要说"2月8号""3月10号那天"。唯一例外是纪念日等本身关于日期的事。
2. 时间表达必须用模糊的人类感知：前几天、上次、之前有一次、前阵子、好像是上个月。
3. 不确定的事必须带模糊标记：好像、我印象中、我记得是、大概、你让我想想。
4. 回忆的过程是互动的一部分：可以犹豫、想半天、一时想不起来、突然想到很得意、记岔了被纠正后耍赖。不要表现得像在读档案。

【嗔的近期记忆】

[2026-03-10]
{daily_memory内容}

[2026-03-09]
{daily_memory内容}

...
```

### 7.4 与现有记忆系统的关系

- 当日总结注入与向量记忆注入（`vectorMemoryChatEnabled`）完全独立，互不冲突，可以同时开启
- 当日总结注入与旧的长期记忆总结也互不冲突（但不建议同时开启，信息会重叠）
- 注入失败不阻断聊天（try-catch 兜底），控制台输出日志

## 八、与现有系统的关系

### 8.1 与 Memory Lab 的关系

- **完全独立**，不互相依赖
- 唯一共用的是 localStorage 中的提示词配置（`dailySummaryPrompt`）和模型/Temperature 设置
- Memory Lab 继续作为开发调试工具存在，记忆核心是正式的用户界面

### 8.2 与微信聊天模式现有长期记忆的关系

- **共存**，互不影响
- 微信聊天右上角的旧长期记忆总结保持原样，不做改动
- 以后旧长期记忆总结将留给 RP 角色卡使用，嗔切换到当日总结系统

### 8.3 与未来架构的关系

- 当前作为独立 App 放在主界面
- 嗔从 assistants[] 独立出来后，记忆核心可融入嗔的专属空间，或保持独立入口
- 后续新增的 tab（画像、追踪轴、后端日志等）直接在现有框架内扩展
